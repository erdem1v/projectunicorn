# Project Unicorn — Current State Audit (2026-05-16)

> Read-only inventory pass. No code, scene, config, or git state was modified during this audit. Source documents cross-referenced: `docs/PROJECT_SPEC.md` (design intent) and `docs/TECH_SPEC.md` (architectural rules, including the Decision Log through 2026-05-16).

---

## Section 1 — Executive summary

Project Unicorn today is an early skeleton with a thin functional core. The runtime spine that TECH_SPEC §6 and §8 describe — eight autoload singletons, an EventBus signal hub, a continuous time loop with speed control, and a fixed-order daily tick — is fully wired and behaving correctly. The 6-step onboarding flow reaches the `GameShell` shell, which paints a TopBar HUD, a vertical 8-tab rail, a `[PRODUCT / Content coming]` center placeholder, a right-side info column, and a bottom news ticker, with a modal layer reserved for events. Four of the nine daily tick slots are live (HR, Sales, Finance, Events); the remaining five are stub `pass` bodies in the correct positions, ready to be filled without reordering.

The functional core that does exist is shallow but architecturally clean. `GameState` owns canonical run state (cash, MRR, brand, reputation, day, hour, phase, plus identity fields) and every mutating setter emits a matching `EventBus` signal, so the UI updates one-way exactly as TECH_SPEC §6.2 prescribes. `HRSystem`, `SalesSystem`, and `FinanceSystem` are pure-logic `RefCounted` modules with only `static` methods — they read autoload state, push aggregates back through autoload setters, and never touch a node. `EventManager` loads three debug reactive events from disk at boot, evaluates eligibility (day, phase, cash/brand/reputation thresholds, subgenre, random, cooldown, one-shot), queues by priority, and hands the active event to `main.gd` via `modal_requested` for mounting into the `ModalLayer`. The full vocabulary works end-to-end against placeholder content.

Content and presentation are where the gap to PROJECT_SPEC is widest. Every JSON file under `data/` is a `ev_debug_*` reactive event tagged `[DEBUG]` in the title and body; the other six content directories (`events/industry`, `events/scandals`, `characters`, `companies`, `techtree`, `news`) are `.gitkeep`-only. The onboarding offers one real Origin (Self-Made) and two locked stubs (Heir, Corp Refugee — `ÇOK YAKINDA`), two real Subgenres (AI, SaaS) and one locked (Social), eight trait cards whose descriptions are still `[DEBUG]` placeholders, and a mentor intro modal whose body text ends `[DESIGNER REVIEW — placeholder copy]`. `themes/master_theme.tres` is an empty shell; `scripts/theme/ui_tokens.gd` holds working-placeholder color, font-size, and glyph constants per the 2026-05-16 Decision Log. `localization/` contains only a `.gitkeep` — no CSV, no TR/EN split has been initialized — and every visible string is hardcoded Turkish-with-English-fragments. `SaveManager` and `Settings` autoloads are empty stubs.

Larger swathes of PROJECT_SPEC are not implemented at all. There is no Product build system, no R&D tech tree, no prospect spawn / pitch dialogue, no hire flow, no credit ladder, no scandal pipeline, no competitor AI, no VC pitch sequence, no term-sheet cinematic, no endings evaluation, no audio. The phase field can be written but no transition logic moves it. The `tabs/` and `desk/` scene directories are placeholder-only, so the eight tabs in the left rail have no swappable views to mount in the center viewport. The visible game today is approximately the onboarding flow, the GameShell layout shell with hardcoded preview values, and the debug event modal driven by three placeholder JSONs.

The architecture, however, is healthy and consistent with TECH_SPEC. Naming follows §12 (snake_case scripts, PascalCase scenes, `class_name` capitalization on data models), the autoload signal pattern of §6.2 is observed with one deliberate exception during run initialization, system purity per §8.3 is preserved with no scene-node leakage, the tick order matches §8.2 verbatim, and the Decision Log entries from 2026-05-14 through 2026-05-16 are visible in the code (`canvas_items` stretch, business-day-start 09:00 anchor, calendar anchor to Wed Jan 1 2025, EventManager pipeline shape, 8-tab `ops/rnd/personal/events` overhaul, working palette constants). The root `CLAUDE.md` is the largest single piece of stale documentation — its "no scripts yet" / "skeleton not yet created" claims have been overtaken by every commit since. A reader of this audit should leave with the picture of a build whose plumbing is correct and over-engineered for its current content load, waiting for designed content and feature systems to fill it.

> **Teaching note — Autoload signal pattern.** TECH_SPEC §6.2 codifies that every UI-visible state mutation must emit a signal and that singletons never reference scene nodes. The autoloads largely **comply**: every setter that mutates a UI-visible field fires the matching EventBus signal (`GameState.set_cash` emits `cash_changed` + `runway_recalculated` at `game_state.gd:39-40`), and no autoload imports a scene path. The one **deliberate drift** is `GameState.initialize_run` (`game_state.gd:103-134`) plus `CharacterRegistry._seed_debug_characters` / `ensure_mentor`, which write fields directly — the comments mark this as intentional because no listeners are wired during onboarding/boot, so emitting would land in the void. The Godot concept underneath is the **autoload singleton** plus its **signal** decoration: a class registered in Project Settings → AutoLoad runs before any scene and remains globally addressable, and `signal foo(...)` declarations expose connect/emit machinery the engine wires implicitly.

### 3.2 FinanceSystem (`scripts/systems/finance_system.gd`)

- **Tick:** `static func daily_tick() -> void` — `finance_system.gd:39`
- **State read:** `CharacterRegistry.get_total_monthly_salaries()` (`finance_system.gd:43`); `GameState.daily_burn` (`:48`); `GameState.mrr` (`:52`); `GameState.cash` (`:56`)

### 3.3 SalesSystem (`scripts/systems/sales_system.gd`)

- **Tick:** `static func daily_tick() -> void` — `sales_system.gd:22`
- **State read:** `CustomerRegistry.get_total_mrr()` (`sales_system.gd:23`); `GameState.mrr` (`:24`)
- **State written:** `GameState.set_mrr(total_mrr)` (`sales_system.gd:25`)
- **Indirect signal emissions:** `mrr_changed` + `runway_recalculated` (via `set_mrr`)
- **Notable patterns:** smallest of the three — single aggregation+push. Same redundant-signal guard pattern. Debug print at `sales_system.gd:28` is the only side effect outside the setter call.

> **Teaching note — System purity.** The three live systems are **pure logic** per TECH_SPEC §8.3: each `extends RefCounted` with only `static` methods, no scene/Node references, no signals of their own, and every write routes through an autoload setter that emits on EventBus. No leaks were observed. The borderline case is `FinanceSystem.burn_breakdown` (`finance_system.gd:27`), a `static var` holding module-level mutable state — but it is still pure (no scene), saveable, and exposed only through `set_burn_category` / `get_burn_breakdown`. All three respect the one-way dataflow rule. The Godot concept is **`RefCounted`** (memory-managed via reference counting, no scene tree needed) — the right base class for stateless logic helpers; if a system later needed cross-day state spanning ticks, an autoload pattern would be the natural promotion path.

The build's largest single piece of stale documentation is `project-unicorn/CLAUDE.md`. Its "Current Project State" block (lines 57-67) and "Next Step" / "What this project is NOT (yet)" blocks (lines 69-80) describe a greenfield repo — "No scenes, no scripts, no data, no theme, no autoloads, no localization files yet" (line 64), "TECH_SPEC §4 directory layout ... not yet created" (line 65), "Skeleton planning ... These claims were accurate at the time of writing but have been overtaken by every commit since; the Decision Log in TECH_SPEC §20 from 2026-05-14 through 2026-05-16 is the actual source of truth on current state. Three distinct placeholder vocabularies coexist in the content (`[DESIGNER REVIEW]` on onboarding card copy, `[DEBUG]` / `[DEBUG content]` on trait descriptions and event JSON, `[NOT IMPLEMENTED]` on one event choice gated by an impossible `phase: 99`); a single sweep to standardize the marker would reduce search cost during the content phase. The Frank Köseoğlu / Operating Partner identity is hardcoded as literal strings in three places (`MentorIntroModal.tscn`, `RightPanel.tscn`, and the body text of `ev_debug_003_cash_warning.json`'s `character_id`), and the right panel's customer/rival/upcoming-event rows are sample data baked into the scene file rather than wired through `CustomerRegistry` / a hypothetical `RivalRegistry` / `EventManager.get_upcoming()` — these will paint over correctly when the wiring scripts run, but the literals sit in version control until then.

Architectural patterns are converging on a clear shape worth naming. Pure-logic systems all `extends RefCounted` with `static` methods, autoloads all expose mutating setters that emit on EventBus and treat direct field writes as a deliberate exception during boot/init, every registry uses `get_<thing>(id)` instead of `get(id)` to dodge `Object.get` collision, every signal handler in scenes uses `_on_<signal_name>` naming, and the `_seed_debug_*` private methods are the canonical place for prebuilt content. A `.gitkeep` in `scripts/data_models/` still exists despite the directory now holding four real models, signaling the skeleton was committed before content and never tidied; a stray `node_2d.tscn` at the project root sits outside the documented `scenes/` tree and looks like a Godot editor scratch file. None of these are blocking, but together they sketch the natural cleanup-and-consolidation pass that the post-Bootstrap content phase will want to run before further scenes pile on top.
